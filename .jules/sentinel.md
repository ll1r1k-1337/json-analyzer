## 2024-05-24 - [Unbounded Allocation OOM DoS]
**Vulnerability:** The binary token reader allows malicious input to request arbitrarily large memory allocations. The binary format uses up to 4-byte length prefixes (uint32) for payloads like Strings and TypedArrays, which directly dictate the number of bytes allocated in `Buffer.allocUnsafe` and `Buffer.alloc` via `FileReader.read` and `BinaryTokenReader.readBytes`. An attacker can manipulate these length fields to attempt massive memory allocations (up to 4GB per read), resulting in an immediate Out-Of-Memory (OOM) crash and Denial of Service (DoS) for the application.
**Learning:** This existed because length values parsed directly from binary streams were implicitly trusted and used without boundary checking, assuming data integrity.
**Prevention:** Always validate length fields from untrusted binary/external inputs against a safe maximum threshold (e.g., `MAX_SAFE_ALLOCATION`) before using them to dictate memory allocations or buffer sizes.
