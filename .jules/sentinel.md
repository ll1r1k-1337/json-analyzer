## 2024-05-20 - Unbounded String Buffering / OOM Risk in Binary Writer
**Vulnerability:** The `BinaryTokenWriter` did not enforce limits on the number of unique strings (`maxUniqueStrings`) or the maximum string table size (`maxStringTableBytes`), allowing a large stream of arbitrary JSON keys/strings to exhaust memory and cause a DoS (OOM attack). Furthermore, `value.length` was incorrectly used instead of `Buffer.byteLength` for calculating total bytes when looking up existing strings in the index, which could under-report memory usage for non-ASCII text.
**Learning:** Even though `BinaryTokenWriter` is a streaming parser, its internal "String Table" buffer holds strings in memory until finalization. Because of this, large continuous allocations in a streamed file bypass normal stream pressure constraints. Memory-tracking stats must always use proper `Buffer.byteLength` instead of `.length` to accurately reflect memory consumption of UTF-8 encoded text.
**Prevention:** Implement limits (`maxUniqueStrings` and `maxStringTableBytes`) directly where memory allocation occurs (e.g., `registerString`). Ensure string size metrics always use `Buffer.byteLength(value, 'utf8')` instead of naive `.length` to avoid discrepancies between JS string length and actual buffer requirements.
