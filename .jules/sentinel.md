## 2024-05-24 - DoS via Unbounded Buffer Allocation in Binary Reader
**Vulnerability:** The `BinaryTokenReader` relies on reading length-prefixed data from the input source. For payloads like Strings, Numbers, and TypedArrays, the format specifies a length up to 4 bytes (uint32), which allows the attacker to specify an allocation up to 4GB. Reading an allocation size directly and immediately requesting a buffer of that size could lead to Out-Of-Memory (OOM) / Denial of Service (DoS) attacks if a malicious file specifies an excessively large length.
**Learning:** Even though the underlying file reader limits chunk sizes or implements buffering, the `readBytes` method requests the full payload length at once. This effectively bypasses buffering mechanisms when requesting tokens with massive apparent lengths, forcing a single large memory allocation.
**Prevention:** Implement a maximum safe allocation limit (`MAX_SAFE_ALLOCATION`) that caps the maximum size of a single buffer that `readBytes` is allowed to request. A default of 512MB is a reasonable limit that allows for legitimate massive payloads (like large TypedArrays or Base64 blobs) while providing a hard stop against OOM exploitation.
