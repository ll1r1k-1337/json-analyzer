## 2026-03-13 - Fix Out-of-Memory (OOM) DoS in Token Parsing
**Vulnerability:** The binary token parser (`BinaryTokenReader`) accepted completely unsanitized unverified token lengths (up to 4GB via `readUInt32LE`) from the token stream, passing these lengths directly to `Buffer.allocUnsafe` in the `FileReader` block. A maliciously crafted binary file could inject lengths of 4GB, causing a hard process crash (Assertion failed: `args[3]->IsInt32()`) or triggering `FATAL ERROR: Ineffective mark-compacts near heap limit Allocation failed - JavaScript heap out of memory`, essentially acting as an unhandled Denial of Service attack vector.
**Learning:** Directly passing unbounded lengths extracted from untrusted binary/external payloads directly into dynamic allocation calls (`Buffer.alloc`, `Buffer.allocUnsafe`) creates a vector for process disruption via Memory exhaustion/crashing.
**Prevention:** Bound input lengths before passing them to internal allocations, establishing maximum sensible limits (e.g., `MAX_SAFE_ALLOCATION`) for variable-length items like strings or arrays parsed from binary input formats.
