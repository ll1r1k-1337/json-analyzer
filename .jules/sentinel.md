## 2024-05-23 - Prevent Out-of-Memory (OOM) via Unbounded Token Lengths
**Vulnerability:** `BinaryTokenReader` attempts to allocate unbounded token lengths for `Number` and `TypedArray` tokens without validating against `MAX_SAFE_ALLOCATION`.
**Learning:** Even if `FileReader` bounds allocations by file size/chunks, unbounded allocation attempt logic was missing the explicit `MAX_SAFE_ALLOCATION` limit explicitly. We must defend against maliciously crafted files requesting very large array sizes, mitigating DoS via memory exhaustion. Furthermore, file readers must bounds-check chunk lengths directly against the actual remaining unread file size on disk (`Math.min(length, Math.max(0, this.size - offset))`).
**Prevention:** Always validate parsed length fields from untrusted data against strict, system-imposed upper bounds (like `MAX_SAFE_ALLOCATION`) *before* executing internal I/O or allocating buffers, and strictly bounds-check all physical I/O allocations against file-size.
